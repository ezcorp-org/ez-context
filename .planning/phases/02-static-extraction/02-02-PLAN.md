  <name>Task 1: CI config and project structure extractors</name>
  <files>
    src/extractors/static/ci.ts
    src/extractors/static/project-structure.ts
  </files>
  <action>
    **ci.ts** (`ciExtractor`):
    - Use globby to find CI config files: `.github/workflows/*.yml`, `.github/workflows/*.yaml`, `.gitlab-ci.yml`
    - If no CI files found, return `[]`
    - Use `import { load as yamlLoad } from "js-yaml"` to parse each YAML file
    - For GitHub Actions: traverse `jobs.*.steps[].run` to extract shell commands
    - For GitLab CI: traverse top-level job keys (skip `stages`, `variables`, `include`, `default`), look for `script` arrays
    - Filter commands to known build/test/lint patterns:
      - Build: substring match for `build`, `compile`, `tsc`, `tsdown`
      - Test: substring match for `test`, `vitest`, `jest`, `pytest`, `cargo test`, `go test`
      - Lint: substring match for `lint`, `eslint`, `biome`, `clippy`, `ruff`
    - For each matched command, create entry: category "stack" (build/lint) or "testing" (test), pattern `CI command: {command}`, confidence 0.9
    - Evidence: `[{ file: relativePathOfCiFile, line: null }]`
    - Store ALL raw `run:` commands in metadata.rawCommands for inspection
    - Wrap YAML parsing in try/catch -- malformed YAML returns `[]` for that file, not a thrown error

    **project-structure.ts** (`projectStructureExtractor`):
    - Use globby to check test file patterns (from RESEARCH.md Pattern: Test Pattern Detection):
      - `**/*.test.{ts,tsx,js,jsx}` -> "co-located, *.test.ts style"
      - `**/*.spec.{ts,tsx,js,jsx}` -> "co-located, *.spec.ts style"
      - `test/**/*.{ts,tsx,js,jsx}` -> "test/ directory"
      - `tests/**/*.{ts,tsx,js,jsx}` -> "tests/ directory"
      - `__tests__/**/*.{ts,tsx,js,jsx}` -> "__tests__/ directory"
    - Use ALWAYS_SKIP from `../../utils/fs.js` as ignore patterns, plus `gitignore: true`
    - For each pattern with matches, create entry: category "testing", pattern `Test files in {location} ({style})`, confidence `min(0.95, 0.5 + count * 0.05)`
    - Evidence: first 5 matched files as evidence refs
    - Metadata: `{ testFileCount, location, style }`
  </action>
  <verify>
    - `bun run build` succeeds
    - `bun run lint` passes
    - Both extractors compile and export correctly
  </verify>
  <done>
    CI extractor can parse GitHub Actions YAML and extract build/test/lint commands. Project structure extractor detects test file patterns with location and count.
  </done>

<task type="auto">
  <name>Task 2: Naming convention and import pattern extractors (ts-morph)</name>
  <files>
    src/extractors/code/naming.ts
    src/extractors/code/imports.ts
  </files>
  <action>
    **naming.ts** (`namingExtractor`):
    - Use `listProjectFiles` from `../../utils/fs.js` with extensions `["ts", "tsx", "js", "jsx"]`
    - If no files found, return `[]`
    - Cap files at `ctx.options?.maxFilesForAst ?? 200` (take first N after sort)
    - Create ts-morph `Project` with `{ compilerOptions: { allowJs: true, noEmit: true } }` -- NO tsconfig
    - Add source files via `project.addSourceFilesAtPaths()`
    - Implement `classifyCase(name: string)` returning `"camelCase" | "PascalCase" | "snake_case" | "UPPER_SNAKE_CASE" | null`
      - Return null for names shorter than 4 chars (filter noise per RESEARCH.md pitfall 6)
      - UPPER_SNAKE_CASE: `/^[A-Z][A-Z0-9_]{3,}$/`
      - PascalCase: `/^[A-Z][a-zA-Z0-9]+$/`
      - snake_case: `/^[a-z][a-z0-9_]+$/` AND contains `_`
      - camelCase: `/^[a-z][a-zA-Z0-9]+$/` AND contains uppercase
      - Otherwise return null (unclassifiable)
    - Count cases per entity type: functions (getName), variables (getName), classes (getName)
    - For each entity type with >= 5 total identifiers, find dominant case
    - Confidence: `min(0.95, dominantCount / total)`, only emit if >= 0.6
    - Entry: category "naming", pattern `{entityType} use {case} naming`, metadata with counts and sampleSize
    - Evidence: `[{ file: "src/**/*.ts", line: null }]`

    **imports.ts** (`importsExtractor`):
    - Same file discovery + ts-morph setup as naming (use listProjectFiles, cap at maxFilesForAst)
    - For each source file, get import declarations via `sourceFile.getImportDeclarations()`
    - Categorize each import:
      - Relative: `imp.isModuleSpecifierRelative()` returns true
      - External: non-relative (node_modules or alias)
    - Count relative vs external imports across all files
    - Detect barrel file usage: for relative imports, check if `imp.getModuleSpecifierSourceFile()` resolves and the resolved file has only re-exports (exportDeclarations.length > 0 AND no function/class/variable declarations). Count barrel imports.
    - Detect path alias usage: check for non-relative imports that start with `@/` or `~/` or match known alias patterns. Or check tsconfig paths from the tsconfig extractor results (but simpler: just check if the import specifier starts with common alias prefixes).
    - Create entries:
      - If barrel files detected: category "imports", pattern "Uses barrel file (index) imports", confidence based on ratio, metadata with barrelCount
      - Import organization: category "imports", pattern describing ratio (e.g., "Predominantly relative imports" or "Mix of relative and external imports"), confidence based on sample size
      - Path aliases: category "imports", pattern "Uses path aliases (@/ prefix)", confidence 1.0 if detected
    - Evidence: `[{ file: "src/**/*.ts", line: null }]`

    IMPORTANT for both:
    - Do NOT call `project.resolveSourceFileDependencies()` (see RESEARCH.md anti-pattern)
    - Handle the case where `getModuleSpecifierSourceFile()` returns undefined gracefully (just skip that import for barrel detection)
    - Both extractors share ts-morph setup logic -- consider extracting a shared helper `createTsMorphProject(ctx)` into a small utility, or just duplicate the 5 lines in each file (DRY vs coupling tradeoff -- duplicating is fine for 5 lines)
  </action>
