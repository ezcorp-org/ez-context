<task type="auto">
  <name>Task 1: Error handling semantic extractor</name>
  <files>src/extractors/semantic/error-handling.ts, test/extractors/semantic/error-handling.test.ts</files>
  <action>
Create `src/extractors/semantic/error-handling.ts` implementing the `Extractor` interface.

The extractor must:
1. Call `createBridge(ctx.projectPath)` and check `bridge.hasIndex()` -- return `[]` if no index
2. Issue 3 targeted search queries:
   - `"error handling try catch throw exception"` (k: 30) -- general error handling
   - `"Result Ok Err return error value"` (k: 20) -- functional/Result types
   - `"custom error class extends Error"` (k: 20) -- custom error classes
3. Merge all chunks, deduplicate by `chunk.file` using a Map (key=file, value=concatenated chunk text)
4. Classify each file's content against these patterns:
   - **try/catch**: `/\btry\s*\{/` AND `/\bcatch\s*\(/` in same file
   - **Result/Either**: `/\bResult<|\bOk\(|\bErr\(|\bisOk\b|\bisErr\b|\bneverthrow\b/`
   - **Custom error classes**: `/class\s+\w+Error\b|\bnew\s+\w+Error\(/`
   - **Error boundaries**: `/\bErrorBoundary\b|\bcomponentDidCatch\b/`
5. For each pattern found in >= 2 distinct files, emit a ConventionEntry with:
   - `category: "error_handling"`
   - `pattern`: descriptive string (e.g., "try/catch imperative error handling")
   - `confidence`: `Math.min(0.95, 0.5 + (matchingFiles / totalUniqueFiles) * 0.45)`
   - `evidence`: up to 5 file references from the matching set
   - `metadata`: `{ style: "try-catch" | "result-type" | "custom-error-class" | "error-boundary", fileCount: N }`

IMPORTANT: Never call `bridge.embed()` -- it throws. Only use `bridge.search()`.

Create `test/extractors/semantic/error-handling.test.ts`:
- Mock `createBridge` using `vi.mock("../../src/core/ez-search-bridge.js", () => ({ ... }))` factory pattern
- Test cases:
  1. Detects try/catch pattern when search returns chunks with try/catch from 3+ files
  2. Detects Result type pattern from functional-style chunks
  3. Returns [] when hasIndex returns false (no index)
  4. Returns [] when search returns empty results
  5. Skips patterns found in only 1 file (below 2-file minimum)
  6. Confidence capped at 0.95 even with many matching files
  </action>
  <verify>Run `bun run test -- test/extractors/semantic/error-handling.test.ts` -- all tests pass</verify>
  <done>Error handling extractor detects try/catch, Result types, custom error classes from mocked search results; returns [] gracefully when no index; confidence scales with file count and caps at 0.95</done>
</task>

<task type="auto">
  <name>Task 2: Architecture semantic extractor</name>
  <files>src/extractors/semantic/architecture.ts, test/extractors/semantic/architecture.test.ts</files>
  <action>
Create `src/extractors/semantic/architecture.ts` implementing the `Extractor` interface.

The extractor uses TWO complementary signals:

**Signal 1 -- Directory structure scan (deterministic, no search needed):**
1. Call `listProjectFiles({ cwd: ctx.projectPath, extensions: ["ts", "js", "tsx", "jsx", "py", "rb", "go", "rs"] })` to get all source files
2. Extract top-level directory names under `src/` (or project root if no src/). Use `path.split("/")` to get the first 1-2 directory segments.
3. Classify directory names:
   - **MVC**: >= 2 of `models/`, `views/`, `controllers/`, `routes/` (case-insensitive, with or without trailing s)
   - **Feature-based**: files under directories matching `/\/(features?|modules?|pages?)\//i` with >= 5 files
   - **Layer-based**: >= 2 directories matching DDD (`domain`, `application`, `infrastructure`), hexagonal (`service`, `repository`, `handler`, `usecase`), or clean arch (`core`, `data`, `presentation`)

**Signal 2 -- Semantic search confirmation (adds evidence):**
1. Check `bridge.hasIndex()` -- if no index, use directory signal only
2. If index exists, issue 1 search query: `"model view controller route handler service repository"` (k: 20)
3. Use search chunks as additional evidence references only

**Output:**
- Emit a single ConventionEntry with:
  - `category: "architecture"`
  - `pattern`: e.g., "MVC architecture pattern" or "Layer-based architecture"
  - `confidence`: 0.7 for directory-only detection, 0.85 if search confirms
  - `evidence`: directory paths + search chunk file refs (up to 5)
  - `metadata`: `{ architecturePattern: "MVC" | "feature-based" | "layer-based", layers: ["src/models", "src/views", ...] }`
- If no pattern detected, return `[]`

IMPORTANT: The `metadata.architecturePattern` and `metadata.layers` fields are used by Plan 02's `populateArchitectureInfo` to set `registry.architecture.pattern` and `registry.architecture.layers`.

Create `test/extractors/semantic/architecture.test.ts`:
- Mock both `createBridge` and `listProjectFiles`
- Test cases:
  1. Detects MVC from directory structure (models/, views/, controllers/)
  2. Detects feature-based from directories with features/moduleA/, features/moduleB/
  3. Detects layer-based from directories (domain/, application/, infrastructure/)
  4. Returns [] when no architecture pattern matches
  5. Works without index (directory-only mode returns confidence 0.7)
  6. Confidence increases to 0.85 when search confirms pattern
  7. metadata.layers populated with detected directory names
  </action>
  <verify>Run `bun run test -- test/extractors/semantic/architecture.test.ts` -- all tests pass</verify>
  <done>Architecture extractor detects MVC/feature-based/layer-based patterns from directory structure; optionally confirms via search; emits metadata.architecturePattern and metadata.layers for pipeline consumption</done>
</task>
